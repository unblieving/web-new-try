# 001：课程目录

> 状态：实现中（待补齐空状态与验收测试）  
> 关联事项：课程模板首个端到端示例

## 目标

学习者可以打开首页，确认前后端连通，并看到由 SQLite 保存的课程模块。

## 用户故事

作为一名学习者，我希望查看课程模块及简介，以便理解课程覆盖范围。

## 范围

- 提供服务健康状态和课程列表 HTTP API。
- SQLite 首次初始化时提供三条可展示的课程示例数据。
- 首页明确呈现课程请求的加载、成功、空结果和错误状态。
- 首页的链接与状态信息具有可理解的语义，键盘用户能看到链接焦点。

## 非目标

- 创建、编辑或删除课程的用户界面。
- 课程搜索、详情、认证与授权。
- 数据库 schema 迁移和端到端浏览器测试框架选型。

## 业务规则

- **BR-01**：健康检查返回固定的正常状态、服务标识和 ISO 8601 时间戳。
- **BR-02**：课程列表按 `id` 升序返回；无记录时返回成功响应和空数组。
- **BR-03**：首次初始化发现课程表为空时写入三条示例课程；后续启动不得重复写入。
- **BR-04**：请求进行中、成功且非空、成功但为空、请求失败是四种不同的用户可见状态。

## Contract 影响

- 结论：已有 Contract，本次仅整理 Spec，不改变 HTTP 行为。
- 理由或影响摘要：前端与后端共同依赖健康检查和课程列表；权威字段及状态码见 OpenAPI。
- OpenAPI operation：`GET /api/health`（`getHealth`）、`GET /api/courses`（`listCourses`）。
- 迁移 / 废弃安排：不适用。

## 验收标准

- **AC-01**：当请求 `GET /api/health` 时，响应为 `200`，并包含 `status: "ok"`、服务标识和合法的 ISO 8601 时间戳。
- **AC-02**：给定多条课程数据，当请求 `GET /api/courses` 时，响应为 `200`，`data` 中课程按 `id` 升序排列。
- **AC-03**：给定首次启动且课程表为空，当服务初始化完成后，课程列表恰好包含三条示例课程；再次启动不会重复插入。
- **AC-04**：给定课程请求尚未完成，当用户打开首页时，课程区域显示可识别的加载状态。
- **AC-05**：给定课程请求成功且列表非空，当首页完成加载时，每门课程显示标题和简介，API 状态显示为已连接。
- **AC-06**：给定课程请求成功但 `data` 为空，当首页完成加载时，课程区域显示明确的空结果说明，不继续显示加载骨架。
- **AC-07**：给定健康检查或课程请求失败，当首页完成请求时，页面显示错误状态和不包含敏感信息的处理提示。
- **AC-08**：当键盘用户聚焦“检查 API”链接时，链接具有可见焦点；课程区域及连接状态具有可被辅助技术理解的名称或状态语义。
