# Spec 编写规范

`specs/` 用于记录人与 AI 共同遵守的可执行意图：为什么做、用户能完成什么、本次边界是什么、怎样证明完成。`contracts/` 记录跨端 HTTP 契约，代码负责具体实现。

好的 Spec 描述外部可观察行为，不预设代码落点。除非某项实现约束本身就是需求，否则不要指定文件、组件、类、SQL 或内部调用方式。

## 何时必须写

在实现前新增或更新 Spec，只要改动符合以下任一项：

- 新增或改变用户可见的功能、页面、交互流程或业务结果；
- 新增或改变 HTTP API、持久化事实，或前后端之间的协作行为；
- 改变校验、排序、分页、错误、权限、并发或其他业务规则；
- 属于需要多处协作、存在多种边界场景或需要独立验收的实质功能。

以下改动可免写新的 Spec：

- 仅修正文案、拼写、注释或文档排版；
- 不改变外部行为和 Contract 的格式化、机械重构或测试整理；
- 已有 Spec 已明确期望行为的缺陷修复。此时在缺陷记录中引用原 Spec 和 AC；若期望行为改变，必须先更新原 Spec。

无法确定是否影响用户行为、Contract 或业务规则时，按“必须写”处理。

## 编号与命名

- 文件名使用 `NNN-kebab-case.md`，例如 `002-campus-marketplace.md`；`NNN` 为三位递增编号，slug 使用小写英文和连字符。
- 标题使用 `# NNN：中文功能名`，编号与文件名一致。
- 新编号取现有最大编号加一；编号一经使用不重排、不复用。`README.md` 与 `_template.md` 不参与编号。
- 一个 Spec 对应一项可独立验收的用户能力。范围过大时按用户结果拆分，不按前端、后端或文件目录拆分。

## 必备内容

每份 Spec 从 [`_template.md`](./_template.md) 复制，并完整回答：

1. **目标**：一句话说明用户或业务最终获得的结果，不写技术方案。
2. **用户故事**：使用“作为……我希望……以便……”明确角色、能力和价值。
3. **范围 / 非目标**：分别列出本次必须交付和明确不做的内容，防止隐式扩展。
4. **业务规则**：写清条件、结果与失败语义；规则应能转化为验收或测试。
5. **Contract 影响**：写明 `无 / 新增 / 变更 / 废弃`。若有影响，列出 Method、Path、兼容性和待同步的 OpenAPI operation；字段与响应的权威定义仍在 `contracts/openapi.yaml`。
6. **验收标准**：从 `AC-01` 起连续编号，每条只表达一个可观察结果。
7. **验证映射**：每条 AC 至少映射一个可重复的测试、检查或人工步骤。

## 验收标准写法

优先使用“给定 / 当 / 则”或“条件 → 操作 → 可观察结果”。结果应能从页面、HTTP 响应、持久化结果、日志或测试输出观察，避免“体验良好”“正确处理”“代码已完成”等主观表述。

按改动内容补齐以下场景，并将适用项写成带编号的 AC：

### 前端

- 涉及异步数据时，覆盖**加载、成功、空结果、错误**四态，并说明每种状态用户能看到或执行什么。
- 涉及交互时，覆盖语义化名称或标签、纯键盘可操作、清晰可见的焦点；错误提示需要能与对应输入关联，必要时可被辅助技术感知。
- 涉及布局时，明确需验收的视口和溢出行为；不要用具体组件实现替代用户可见结果。

### 后端

- 有外部输入时，明确缺失、`null`、空字符串、去空格、类型、格式和长度等适用边界。
- 列表或查询接口明确空结果、稳定字段结构、排序依据和分页规则；例如空结果是 `200 []` 还是其他语义。
- 明确成功与失败的状态码、错误结构，以及客户端能否稳定区分不同结果。
- 涉及身份或受限资源时，明确未认证、无权限和资源归属规则；默认拒绝未授权访问。
- 涉及共享可变状态、名额、余额、库存或重复提交时，明确并发结果、事务 / 回滚语义和可验证的不变量。

只有相关场景才需要写入；不适用的项目不要虚构规则。

## Spec 与 Contract

- Spec 定义业务意图和可观察行为；`contracts/openapi.yaml` 是 HTTP 路由、参数、Schema、状态码和错误响应的权威来源。
- Contract 影响为“新增、变更或废弃”时，必须在实现同一批改动中同步 `contracts/openapi.yaml`，并让前端、后端和契约测试保持一致。
- Contract 影响为“无”时也要写出一句理由，例如“仅调整现有响应的页面呈现，不改变 HTTP 请求或响应”。
- 发现 Spec 与 Contract 冲突时先澄清业务语义，再更新文档，不能以当前代码作为默认答案。

## 验证映射

实现前填写计划验证方式，实现后补充命令、用例或证据。可选择静态检查、单元测试、组件测试、API / Contract 测试、端到端测试和人工验收；人工步骤必须可复现。

一条 AC 可以有多种证据，但不能没有证据。仓库级门禁统一运行：

```bash
npm run check
```

## 生命周期

1. **草案**：创建编号，澄清目标、范围、规则、Contract 影响和 AC。
2. **已确认**：相关方确认“做什么、什么算完成”；实质功能在此后开始实现。
3. **实现中**：代码、Contract 和测试围绕同一组 AC 演进。需求改变时先更新 Spec，再继续实现。
4. **已验收**：所有 AC 有通过证据，Contract 与实现一致，完成定义全部满足。
5. **已废弃**：能力不再适用时保留历史文件，注明替代 Spec，不删除或复用编号。

## 完成定义（DoD）

只有同时满足以下条件，Spec 才能标记为“已验收”：

- 目标、范围、非目标和业务规则无未决占位或歧义；
- 所有适用的前端四态、可访问性及后端边界场景均已成为 AC；
- 每条 AC 均有验证映射和通过证据；
- 有 HTTP 影响时，`contracts/openapi.yaml`、前端调用、后端实现和测试已同步；无影响时已记录理由；
- `npm run check` 通过，且人工验收项已记录实际结果；
- 实现没有扩展到非目标；若发生范围变化，Spec 已先更新并重新确认。
